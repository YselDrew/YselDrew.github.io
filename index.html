Місце проходження практики

Компанія «Українські Інформаційні Технології» є розробницьким центром компанії SoftServe і однією з найбільших ІТ-компаній в Україні та Центрально-Східній Європі, що спеціалізується на розробці програмного забезпечення, ІТ-консалтингу та наданні цифрових рішень для бізнесу.

Зміст практики

У рамках проходження виробничої практики було поставлено завдання розробити веб-додаток для базового обліку фінансів.

Проект складається з двох частин: серверної (backend) та клієнтської (frontend). Для реалізації проєкту було використано:

  • NestJS — фреймворк для побудови структурованого серверного застосунку;
  • PostgreSQL — система керування базами даних;
  • TypeORM — ORM для зручної взаємодії з реляційною базою даних;
  • JWT — для аутентифікації користувачів;
  • DTO та class-validator — для валідації вхідних даних.
  • React — фреймворк для побудови клієнтського застосунку.

Проектування бази даних

  • users — зберігає інформацію про користувачів (email, hash пароля, isAdmin)
  • groups — групи для об’єднання рахунків
  • accounts — рахунки з сумою, валютою, описом; прив’язані до userId та groupId
  • categories — категорії витрат/доходів (наприклад: "Продукти", "Зарплата")
  • transactions — фінансові операції з полями:
    • type: income / expense
    • accountId, categoryId
    • сума, нотатка
    • createdAt — дата створення

Всі таблиці мають відповідні зв’язки через зовнішні ключі. Реалізовано каскадне видалення (наприклад, при видаленні рахунку — видаляються пов’язані транзакції).

Дизайн серверної частини

Бекенд API реалізований на основі REST-архітектури з чіткою структурою маршрутів та відповідей.

  • Кожна сутність має власний контролер і сервіс: /users, /accounts, /groups, /transactions, /categories.
  • Маршрути розділені за типом дії:
    • GET — отримання списків або одного елемента
    • POST — створення нового запису
    • PATCH — оновлення даних
    • DELETE — видалення
  • Усі запити до захищених маршрутів вимагають JWT-токен через заголовок Authorization: Bearer.
  • Сервіси містять логіку перевірки userId, щоб користувач міг працювати лише зі своїми даними.
  • Деякі маршрути підтримують пагінацію через параметри page та limit.

Валідація вхідних даних

У застосунку для валідації даних активно використовуються DTO-класи (Data Transfer Object) з декораторами з бібліотеки class-validator.

Валідація виконується автоматично завдяки інтеграції ValidationPipe на глобальному рівні.

Реалізація реєстрації та авторизації

У застосунку реалізована система реєстрації та входу на основі JWT (JSON Web Token).

Після успішного входу або реєстрації сервер видає користувачу JWT-токен, який містить його унікальний userId та email. Токен повертається клієнту та зберігається у localStorage.

Додавання груп, рахунків та категорій

Кожна сторінка побудована за принципом двох колонок: ліворуч розміщено список вже створених записів, а праворуч — форма створення нових.

При створенні нового рахунку користувач вводить назву, суму, валюту, опис та вибирає до якої групи він належить.

Додавання категорій є простішим — достатньо лише вказати її назву. Категорії використовуються при створенні транзакцій і допомагають групувати витрати або доходи.

Робота з транзакціями

користувач може створювати, переглядати та видаляти транзакції, які бувають двох типів: доходи (income) та витрати (expense).

інтерфейс побудовано у вигляді таблиці, де кожна транзакція має такі колонки: тип, категорія, сума з валютою, нотатка та дата створення. у крайньому стовпчику — кнопка видалення.

додавання нової транзакції виконується через модальне вікно.

після створення транзакції баланс відповідного рахунку автоматично змінюється: якщо це витрата — сума віднімається, якщо дохід — додається. те ж саме відбувається у зворотному напрямку при видаленні транзакції.

для зручності реалізовано пагінацію — користувач може переглядати історію своїх транзакцій, перемикаючись між сторінками. дані оновлюються через api з параметрами page і limit.

Система контролю версій

У процесі розробки застосунку активно використовувалася система контролю версій Git. Основна розробка велася у гілці main.

Для хостингу репозиторію використовувався GitHub.
